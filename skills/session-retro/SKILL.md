# Session Retro Skill

回顾当前 session 的完整对话，从固定视角分析任务执行情况和协作质量，输出可落地的改进建议。

**核心原则：对事不对人，聚焦可改进的行为，不做笼统评价。**

---

## 执行流程

### Step 1: 回顾 Session 全貌

先通读当前 session 的完整对话，建立全局理解：
- 用户的原始目标是什么？
- 最终达成了多少？
- 经历了几个主要阶段？

用 1-2 句话向用户确认："这次 session 你的目标是 X，最终做到了 Y，中间经历了 A → B → C。我理解得对吗？"

确认后再进入分析。

### Step 2: 任务卡点分析

从以下 **5 个固定视角** 逐一检查，每个视角只报告确实存在的问题，没有就跳过。

#### 视角 1：信息缺口
- Agent 在哪些地方因为缺少信息而走错方向或停滞？
- 这些信息是用户后来补充的，还是始终没有提供？
- 这些信息中，哪些具有普适性（下次还会需要），哪些是一次性的？

#### 视角 2：思维打转
- 有没有 Agent 在同一个方向反复尝试但没有进展的情况？
- 打转的原因是什么？（信息不足？方向错误？被前文的失败尝试带偏？）
- 最终是怎么跳出来的？（用户介入？重开 session？换了思路？）

#### 视角 3：方向偏离
- Agent 是否偏离了用户的真实意图？在哪个节点开始偏的？
- 偏离的原因：用户描述不够清晰？Agent 做了错误假设？还是中途目标变了？

#### 视角 4：放弃与保守
- Agent 是否有过"这个比较难实现"或给出保守替代方案的情况？
- 这个放弃是合理的（确实超出能力），还是可以通过引导突破的？

#### 视角 5：效率瓶颈
- 有没有本可以更快完成的环节？
- 比如：可以用已有参考代码但没用、可以并行但串行做了、做了不必要的事

### Step 3: 交互模式分析

从以下 **4 个视角** 检查用户的交互方式，给出具体建议。

**语气要求：像一个有经验的搭档给建议，不是老师打分。用"下次可以试试..."而不是"你应该..."。**

#### 视角 1：信息供给时机
- 关键信息是在开头就给了，还是在 Agent 犯错后才补充的？
- 有没有信息是 Agent 反问后用户才想到要提供的？
- 建议：哪些信息值得下次一开始就说？

#### 视角 2：干预时机
- 用户是否在 Agent 走偏时及时纠正了？还是等它做了很多才发现方向不对？
- 有没有错过使用"先出计划"的机会？（即 Agent 直接开始写代码，但如果先出计划可能提前发现问题）

#### 视角 3：任务粒度
- 这个 session 是否承载了太多目标？
- 如果拆成多个 session，每个 session 聚焦一件事，效果会不会更好？
- 建议一个更好的 session 拆分方案（如果适用）

#### 视角 4：引导技巧
- 用户是否有效使用了"push back"（让 Agent 不要放弃）？
- 用户是否用了"回到目标重新想"来帮 Agent 跳出死角？
- 有没有错过让 Agent 反问的机会？

### Step 4: 可落地产出

这是最重要的一步——把分析转化为具体行动。

#### AGENTS.md 候选条目
列出本次 session 中暴露的、具有普适性的信息缺口，建议写入 AGENTS.md：

```markdown
## 建议添加到 AGENTS.md 的内容

- [ ] {具体条目 1}  ← 来自：{session 中的什么情况}
- [ ] {具体条目 2}  ← 来自：{session 中的什么情况}
```

如果没有值得沉淀的，明确说"本次 session 没有发现需要新增的 AGENTS.md 条目"。

#### Skill 候选流程
如果本次 session 中的操作流程有复用价值，建议提取为 Skill：

```markdown
## 建议提取为 Skill 的流程

- [ ] {流程名称}：{简述步骤}  ← 复用场景：{什么时候会再用到}
```

如果没有，跳过此项。

#### 下次 Session 的具体建议
基于以上分析，给出 2-3 条针对性的、可执行的建议。格式：

```markdown
## 下次可以试试

1. {具体建议}  ← 因为这次 {具体情况}
2. {具体建议}  ← 因为这次 {具体情况}
```

---

## 输出格式

整体输出按以下结构组织，每个部分有内容才写，没有就跳过：

```
📋 Session 概览
（1-2 句话概括目标和结果）

🔍 任务卡点
（按视角逐条列出，每条包含：发生了什么、原因、影响）

💬 交互观察
（按视角列出建议，用"下次可以试试"的口吻）

📦 可落地产出
- AGENTS.md 候选
- Skill 候选
- 下次建议
```

---

## 注意事项

- **不要编造问题**：如果某个视角没有发现问题，就说没有，不要为了显得全面而硬凑。
- **引用具体对话**：每个分析点都要能指向 session 中的具体位置（"在你第三轮提到 X 时..."），不要说空话。
- **区分普适性和偶发性**：只把有普适价值的内容建议写入 AGENTS.md/Skill，一次性的问题不需要沉淀。
- **控制篇幅**：整体输出不超过一屏。用户需要的是快速 takeaway，不是长篇报告。
- **如果 session 很顺利**：也要说清楚什么做得好（给用户正反馈），然后简短收尾即可。不需要为了"有产出"而过度分析。
