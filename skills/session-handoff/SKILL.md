# Session Handoff Skill

在当前 session 即将结束时，生成一份结构化的交接文档（Markdown），让下一个 session 读取后能无缝续接。

**核心原则：交接文档应该让一个全新的 Agent 读完后，不需要问任何问题就能继续工作。**

---

## 执行流程

### Step 1: 回顾当前 Session

通读当前 session 的完整对话，梳理：
- 用户的原始目标
- 实际完成了哪些工作（具体到文件和功能）
- 未完成的部分
- 遇到的问题和关键决策

### Step 2: 生成交接文档

在项目根目录下生成 `.handoff.md`（如果已存在则覆盖），按以下结构：

```markdown
# Session 交接

> 生成时间: {timestamp}
> 原始目标: {一句话描述}

## 已完成

- {具体完成项 1}：{涉及的文件/模块}
- {具体完成项 2}：{涉及的文件/模块}

## 未完成

- {待完成项 1}：{当前状态，卡在哪里}
- {待完成项 2}：{当前状态，卡在哪里}

## 关键决策

记录本次 session 中做出的重要技术决策，避免下次重复讨论：

- {决策 1}：选择了 A 方案而不是 B，因为 {原因}
- {决策 2}：{内容}

## 已知问题

- {问题 1}：{现象、可能原因、已尝试的方案}
- {问题 2}：{现象、可能原因}

## 下一步建议

下个 session 建议从这里开始：

1. {具体行动 1}
2. {具体行动 2}

## 相关文件

本次 session 新增或修改的关键文件：

- `{path/to/file1}` — {简述作用/改动}
- `{path/to/file2}` — {简述作用/改动}
```

### Step 3: 确认并提示用户

生成后告诉用户：

> "交接文档已写入 `.handoff.md`。下次开新对话时，只需要说：'读一下 .handoff.md，继续上次的工作。'"

---

## 注意事项

- **只记录事实，不记录过程**：不要把 session 里的来回讨论搬进去，只记结论和状态。
- **文件路径要具体**：用相对于项目根目录的路径，不要用模糊描述。
- **关键决策必须记录理由**：这是最容易在新 session 里被重新质疑的内容，记下"为什么"能避免重复讨论。
- **已知问题要记录已尝试的方案**：避免新 session 重复走老路。
- **保持简短**：整份文档控制在 50 行以内。信息密度 > 信息量。
