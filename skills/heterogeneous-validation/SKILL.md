# 异构平台数值正确性验证

指导在异构平台（CPU → GPU/NPU）模型迁移场景下，如何系统地验证数值正确性并撰写可信的精度报告。

> 完整指南见 `references/full-guide.md`，包含详细代码示例、脚本输出规范和自查清单。

## 核心理念

**“对不对”不是数字对比问题，而是功能等价性问题。**

- CPU 的结果不是“真值”（ground truth），只是“参考值”（reference）
- 两个都不精确的结果做差，得到的是“两种近似之间的距离”而非“误差”
- 正确的问题是：**NPU 上的模型是否保持了原模型在目标任务上的能力？**

| 错误思路 | 正确思路 |
|---------|----------|
| “NPU 和 CPU 的 matmul 差了 1e-3，可以接受吗？” | “这个差异传播到终端后，任务指标还在可接受范围内吗？” |
| “allclose 过了，精度没问题” | “allclose 只说明数字接近，还需任务级指标确认” |
| “CPU 结果是 ground truth” | “CPU 结果是 reference；业务指标才是 truth” |

## 验证金字塔（核心框架）

自顶向下定标准，自底向上排故障：

```
L0 应用级：任务指标是否等价？          ← “对不对”的最终裁判
L1 模型级：输出分布是否等价？          ← 统计等价性
L2 模块级：同输入下子模块输出是否接近？ ← 定位问题模块
L3 算子级：单个 op 输出是否接近？       ← 单元测试 / debug
```

### 核心原则

1. **L0 是唯一的“对不对”裁判。** 任务指标等价 = 迁移正确
2. **L1-L3 是证据和 debug 工具，不是结论。**
3. **阈值自顶向下流动。** L0 任务容忍度 → L1 分布差异容忍度 → L2/L3 数值阈值
4. **四层都要做，各有分工。**

### 每层角色速查

| 层级 | 回答的问题 | 典型工具/指标 | 报告作用 |
|------|------------|------------|----------|
| **L0 应用级** | 模型还能用吗？ | benchmark 任务指标 | **结论** |
| **L1 模型级** | 输出分布一致吗？ | N 次运行统计分布对比 | **主要证据** |
| **L2 模块级** | 哪个子模块引入差异？ | 逐层 hook 对比 | **辅助证据/debug** |
| **L3 算子级** | 单个 op 数值行为符合预期？ | allclose、统计量 | **补充证据** |

## 阈值设定方法（按优先级）

> **原则：阈值必须有独立于观测值的依据。** 看到数据后“调”出来的阈值是循环论证。

### 方法 A: 从 L0 任务容忍度反推（最优）

在参考平台上给中间层加受控噪声，观察任务指标变化：
- 找到任务指标变化 < 5% 的最大 eps → 即容忍度
- 若 NPU 逐层 diff 远小于此 eps → 有安全余量

### 方法 B: 多参考平台交叉比较（推荐）

同时在 CPU、CUDA GPU、NPU 上跑，比较所有两两配对的 diff：
- NPU-CPU diff 与 GPU-CPU diff 同一数量级（或仅大 1 个数量级）→ NPU 可信

### 方法 C: 同平台非确定性基线（实用）

在参考平台上多次运行，观察 run-to-run variance：
- NPU-CPU diff 与参考平台自身方差同量级 → 正常浮点波动

### 方法 D: 业界惯例参考（保底）

| 来源 | fp32 典型容忍度 | 说明 |
|------|--------------|------|
| ONNX Runtime | atol=1e-3, rtol=1e-3 | 跨后端一致性测试 |
| TensorRT | >99% 元素 rel_diff < 1% | fp32→fp16 验收 |
| PyTorch | atol=1e-5, rtol=1.3e-6 | 同平台回归测试 |
| 实践经验 | cosine_sim > 0.999 | 整体输出相似度 |

## 扩散模型特殊处理

扩散模型的输出是从分布中采样的，验证逻辑本质不同：

1. **固定噪声对照**: 固定初始噪声和随机种子，对比确定性前向部分
2. **分布等价性检验**: 多次采样（N≥20），用 Mann-Whitney U 检验比较指标分布
3. **不能用单次 allclose**: 单次输出差异混合了“平台差异”和“随机采样差异”

## L3 算子级验证要点

### 硬性判定（必须通过）

无 NaN、无 Inf、shape 一致、dtype 一致。失败 = NPU 实现有 bug。

### 软性判定（记录但不下结论）

1. 记录统计量（含 P90/P99/P99.9，不只有 mean+max）
2. 在 L2 观察差异是否被放大
3. 在 L0/L1 确认是否影响任务指标
4. L0 通过后才能说“L3 差异可接受”

### 推荐统计量

每个算子对比应输出：
- **绝对差异**: max, mean, P50, P90, P99, P99.9
- **相对差异**（排除 |ref|<1e-6 的近零元素）: max, mean, P99
- **异常元素**: NaN 数量, Inf 数量, >1e-3 占比, >1e-2 占比
- **整体相似度**: cosine similarity

> 详细代码见 `references/full-guide.md` 第 7 节。验证脚本见 `scripts/validate_precision.py`。

## 报告结构建议

按验证金字塔组织，L0 放最前：

```
1. 环境信息（硬件、驱动、框架版本）
2. L0 应用级结论（benchmark 指标对比）  ← 读者第一时间看到结论
3. L1 模型级证据（输出分布对比）
4. L2 模块级分析（误差传播）
5. L3 算子级数据（支撑材料）
6. 阈值依据说明（引用方法 A/B/C/D）
7. 性能对比
附录：复现命令、原始数据
```

## “参照什么”的分层回答

当用户问“跑出来的结果对不对，参照什么”时：

1. **第一参照**: 模型标准 benchmark 任务指标
2. **第二参照**: 已验证平台的 diff 量级（GPU-CPU diff 为参照线）
3. **第三参照**: 模型自身数值敏感度（扰动实验）
4. **兆底参照**: 业界惯例（ONNX Runtime/TensorRT）

## 术语规范

- 用“差异 (diff)”而非“误差 (error)”
- 用“参考平台 (reference)”而非“基准/真值 (ground truth)”
- 因果声明需有控制实验支撑
