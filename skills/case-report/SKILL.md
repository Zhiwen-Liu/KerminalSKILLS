# 案例报告生成

从当前项目的 session 对话记录和项目产物中，自动提取关键信息，生成一份商务侧能直接使用的案例报告。

## 使用场景

这个 skill 提供给客户的技术人员使用。他们用 Kerminal 完成了一个项目（迁移、适配、算子开发等），
然后运行这个 skill 一键生成案例报告。报告最终会交给商务/市场团队使用。

因此，使用者通常只会简单地说"帮我生成案例报告"，不会提供额外的上下文或引导。
skill 需要完全自主地扫描项目、分析 session、生成报告，中间不要向用户提问，
除非：1）目录下有多个独立项目需要用户选择（Step 0）；2）询问人工耗时估算（Step 3）。

## 谁会读这份报告

最终读者是商务、市场、管理层——他们大概知道"NPU 迁移"是什么意思，但不关心代码怎么写的。
他们关心的是：

- **能不能做到**：这个任务完成了吗？完成到什么程度？
- **做出了什么效果**：性能数据、测试通过率、精度对比
- **效率有多大提升**：用 Kerminal 花了多久 vs 纯人工要多久，最好能量化
- **人需要参与多少**：介入了几次、做了哪些关键引导
- **可复制性**：类似的项目是不是也能这么做

牢记这个视角，所有技术细节都要翻译成读者能理解的语言，一句话描述里面要总括包含以上关心内容。

## 执行流程

### Step 0: 确定报告目标

在开始收集信息之前，先判断当前目录的结构：

1. 扫描当前目录下的子目录，识别是否存在多个独立项目
   - 判断依据：子目录中包含独立的 `README.md`、`case.toml`、`.git`、`src/` 等项目标志文件
   - 常见场景：kibble 的 `cases/` 目录下有多个案例、一个工作区下有多个项目
2. 如果发现 **多个独立项目**，必须先列出所有项目并询问用户：
   > "当前目录下有以下项目，请问要生成哪个的案例报告？"
   > 1. {项目名 1} — {简要描述}
   > 2. {项目名 2} — {简要描述}
   > ...
   等用户选择后，再进入 Step 1
3. 如果当前目录本身就是一个项目（有 README、源码等），直接进入 Step 1

### Step 1: 收集信息

整个过程不要向用户提问，直接扫描并尽可能多地收集信息。从两个维度收集：

#### 1.1 项目产物扫描

先了解当前项目目录下有什么：

- README、文档（了解项目背景和目标）
- 源代码结构（了解改动范围，不需要读代码细节）
- 测试结果、日志（提取通过率、性能数据）
- 已有的报告或总结文档（避免重复劳动）
- `case.toml`（如果是 kibble 案例，直接提取元数据，这是最高效的信息源）

用 `find` 和 `rg` 快速扫描，重点关注：
- `*.md` 文档文件
- `test*`、`bench*` 相关目录
- `*.log`、`*.csv`、`*.json` 中的结果数据
- `git log --oneline -20` 了解项目开发历史

#### 1.2 Session 记录分析

扫描 `~/.kerminal/sessions/` 下与当前项目 cwd 匹配的 session 文件。

**高效扫描方式**：先只读每个 JSONL 的第一行（`session_meta`，包含 cwd 字段），
匹配到当前项目路径的才深入读取，避免全量扫描所有 session 内容。

```bash
# 只读第一行匹配 cwd，比 grep 全文快得多
for f in $(find ~/.kerminal/sessions/ -name '*.jsonl'); do
  head -1 "$f" | grep -q "<当前项目路径>" && echo "$f"
done
```

从每个 session 的 JSONL 中提取：
- **任务目标**：用户的第一条指令通常说明了要做什么
- **关键里程碑**：测试通过、性能验证成功、问题解决等节点
- **遇到的挑战和解决方式**：体现 Kerminal 的能力
- **耗时信息**：session 的时间跨度（从 timestamp 字段推算）

读 session 时不需要逐条读完，重点看：
- `type: "event_msg"` 中 `user_message` 类型（用户指令）
- `type: "response_item"` 中 `role: "assistant"` 的关键回复
- session_meta 中的时间和项目信息

### Step 2: 信息整合与量化

把收集到的原始信息整理成结构化数据：

#### 项目概况
- 项目名称和一句话描述
- 任务类型（迁移/适配/开发/优化）
- 目标平台和硬件

#### 关键挑战提取

从 session 记录和项目产物中，识别 2-3 个最关键的难点/卡点：
- session 中 Kerminal 反复尝试或用户介入纠正的地方，通常就是难点所在
- 项目文档中提到的"注意事项"、"已知问题"、"踩坑"等内容
- 每个难点要记录：遇到了什么问题、怎么解决的、最终效果如何
- 用非技术语言描述，商务能理解"这事不好做"的程度

#### 成果量化（尽可能用数字说话）

| 维度 | 怎么获取 |
|------|----------|
| 代码改动量 | `git diff --stat` 或 `find` 统计文件数 |
| 测试通过率 | 从测试日志或 session 记录中提取 |
| 性能数据 | 从 benchmark 结果、session 对话中提取 |
| 精度对比 | 从验证日志或对比报告中提取 |
| Kerminal 耗时 | session 时间跨度累加（见下方计算方法） |
| 预估人工耗时 | Step 3 询问用户获取 |

**Kerminal 耗时计算方法**：
1. 取每个 session JSONL 文件的首条和末条 timestamp，算出单个 session 时长
2. 如果同一 session 内相邻消息间隔超过 30 分钟，视为空闲，从有效时长中扣除
3. 多个 session 的有效时长累加，得到总耗时
4. 同时记录 session 跨越的自然天数（用于报告的效率对比）

#### 人工介入分析

这个维度对商务很有说服力——展示"少量关键引导就能完成复杂任务"，
比声称"全自动"更真实也更打动人。

从 session 记录中提取用户的每一次介入，分类统计：

- **介入次数**：用户在 session 中发送的指令总数（排除"继续"、"好的"等无实质内容的确认）
- **关键引导**：真正改变任务方向或解决卡点的指令，每条记录：
  - 用户说了什么（用通俗语言概括，不引用原文）
  - 为什么需要这个引导（Kerminal 遇到了什么情况）
  - 引导后的效果（问题解决了 / 方向调整了）
- **介入类型**：粗略分类，如"方向引导"、"信息补充"、"质量把关"、"环境配置"

判断标准：如果去掉某条用户指令，Kerminal 大概率还能继续正确工作，
那就不算关键引导；如果去掉后会走偏或卡住，那就是关键引导。

**量化原则**：
- 有数据就用数据，不要用"大幅提升"这种模糊说法
- 没有精确数据时，给出合理估算并标注"估算"
- 效率对比是最有说服力的数字，优先确保这个维度有内容

### Step 3: 询问人工耗时

信息收集完成后，**必须**停下来向用户询问一次：

> "如果没有 Kerminal 辅助，纯人工完成这个项目大概需要多久？（比如 X 人天）"

然后等待用户回复：
- 如果用户给出了估算，直接采用，在报告中标注"工程师估算"
- 如果用户表示不确定或不清楚，则根据以下维度自行估算并标注"自动估算"：
  - 代码改动量（文件数、行数）
  - 任务复杂度（迁移/适配/算子开发各有不同基线）
  - session 中遇到的问题数量和解决难度

**这一步必须执行，不能跳过。** 问完并得到回复（或用户明确说不知道）后，再进入 Step 4。

### Step 4: 生成报告

使用 `templates/report.md` 模板生成报告，保存到项目根目录下的 `case-report.md`。

报告写作原则：

1. **开头要有冲击力**：一句话总结放最核心的数字（"3 小时完成原本需要 2 人周的 NPU 迁移"）
2. **紧跟挑战与突破**：核心亮点不能只有效率数字，还要用"难点→解决→结果"的结构展示项目中遇到的关键卡点和怎么突破的。这比纯数字更有说服力——商务需要看到"这事不好做，但我们做到了"
3. **用对比代替绝对值**：不说"推理耗时 0.5s"，说"推理耗时从 GPU 的 0.8s 降到 NPU 的 0.5s"
4. **技术术语要翻译**：不说"算子适配"，说"让模型中的计算模块能在新硬件上运行"
5. **成果要可视化**：善用表格和 ✅ 列表，让人一眼看到重点
6. **少用被动语态**：不说"迁移工作已完成"，说"Kerminal 完成了全部迁移工作"

### Step 5: 自检

生成报告后，用以下清单自检：

- [ ] 报告中没有出现代码片段、函数名、类名
- [ ] 所有数字都有来源（session 记录 or 项目产物），不是编造的
- [ ] 效率对比维度有内容（Kerminal 耗时 vs 人工耗时）
- [ ] "关键挑战与突破"板块至少有 1 条内容，不能为空
- [ ] 一个完全不懂代码的人能读懂每一段
- [ ] 报告长度适中（1-2 页 A4，不超过 500 字正文）
- [ ] 核心亮点在前 3 行就能看到

自检不通过就修改后再输出。

## 特殊情况处理

### 项目未完成

项目可能只完成了一部分（常见情况）。报告应如实呈现完成度，不要美化：
- 在"项目概况"中明确标注完成状态和已完成的比例
- "成果与数据"只报告已验证的部分，未完成的标注"进行中"或"待验证"
- 在"核心亮点"的挑战表格中可以将"未完成的部分"列为待解决项
- 效率对比仍然有价值——展示"已完成部分花了多少时间"

### Session 记录不足

如果找不到足够的 session 记录（比如项目是在其他机器上做的），就主要依赖项目产物，
并在报告开头注明数据来源。效率对比部分如果缺少 session 时间数据，可以根据
代码改动量和复杂度给出保守估算，但要标注为估算值。

### 已有 case.toml 或 summary.md

如果项目中已有 kibble 的 `case.toml` 或 `showcase/summary.md`，
直接从中提取结构化数据作为主要信息源，减少重复分析 session 的工作。

## 输出

最终输出文件：`case-report.md`，放在当前项目根目录下。

生成完成后简要告知用户：报告已生成、文件位置、核心发现（2-3 句话）。
如果有关键信息缺失导致报告某些板块不够充实，一并告知用户可以补充什么信息后重新生成。
