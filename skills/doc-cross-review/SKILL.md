# 迁移文档交叉审查

对硬件迁移项目（CPU → NPU/GPU）的对比验证文档进行系统性交叉审查。
审查目标不是重新运行测试，而是通过文档-代码-文档之间的三角交叉验证发现问题。

## 审查原则

### 1. 文档-代码双向校验

文档中的每个定量声明都必须能在代码中找到对应来源。

```bash
grep -rn 'allclose\|isclose\|atol\|assert.*abs' tests/ scripts/
```

典型矛盾：
- 文档覆盖表将测试文件映射到某章节，但数据实际来自另一个脚本
- 测试代码用 trivial case，验证脚本用真实 case，文档未区分

### 2. 总结-细节一致性校验

| 检查维度 | 典型矛盾 |
|---------|----------|
| 数量 | 总结说“4 项一致”，细节只列2 项 |
| 覆盖 | 总结列5 个组件，细节表6 个 |
| 数值 | 总结表四舍五入后与细节不一致 |
| 结论 | 总结标 PASS，细节数据显示 DIFF |

### 3. 多文档间交叉校验

同一项目多份文档之间数据必须一致：
- 性能数据在两处各出现一次，更新了一处忘了另一处
- 统计口径不一致

### 4. 逻辑自洽性审查

**RNG 差异叙事一致性**:
如果文档解释“模型 A 差异来自 RNG”，那么使用相同 RNG 的模型 B 也应有差异。

**交叉验证链逻辑严密性**:
- 同设备一致性 → 证明 NPU 数值稳定性，不是跨设备精度
- 跨设备算子对比 → 证明基础精度对齐
- 不同维度验证不能混入同一条链

**性能拆分自洽性**:
如果文档声称“固定开销与设备无关”，则 CPU 和 NPU 的固定开销应近似相等。

### 5. 复现命令完整性校验

复现命令必须能完整重现文档中声称的测试条件。检查代码默认值是否与文档声称一致。

### 6. 代码统计口径校验

文档中的文件数、测试数等统计声明，用代码实际计数验证。

### 7. 精度数据 vs 阈值一致性校验

文档声称的 PASS 必须能被实际数据和阈值支撑：
- max_abs vs atol: 超过阈值但标 PASS 时检查是否靠 rtol 补偿
- 文档判定标准必须与脚本实际公式一致

### 8. 文档格式规范性检查

- 不使用非常规 Unicode 符号
- 代码块 \`\`\` 必须成对
- Markdown 表格列数一致

## 工作流程

```
1. 通读所有文档，提取全部定量声明
2. 扫描代码，建立真实覆盖清单
3. 总结表 vs 细节表逐行比对
4. 多文档间数据交叉比对
5. 逻辑链自洽性审查
6. 复现命令 vs 测试条件 vs 代码默认值三方比对
7. 统计数字用代码实际计数验证
8. 按严重程度分级汇总
```

## 输出格式

审查意见保存为 `docs/<原文档名>_review.md`，按严重程度分级：

```markdown
# <原文档名> 审查意见

## 🔴 事实性错误
### 1. <标题>
- **位置**: <文件:行号或章节>
- **问题**: <具体描述>
- **修复建议**: <如何修复>

## 🔴 逻辑矛盾
## 🟡 数据不一致
## ℹ️ 表述可改进
```
